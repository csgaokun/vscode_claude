# HldPlugin 项目概要

> **文档版本**: 1.0

## 一、项目背景

本项目以 Qt Creator 4.13.3 开源 IDE 为技术原型，提取其成熟的插件系统骨架（ExtensionSystem + Aggregation），构建一套**通用的插件化应用框架**。框架的最终形态不是一个 IDE，而是一个**可编辑、可配置、插件驱动的页面型应用**，可用于工业控制、数据展示、设备管理等场景。

选择 Qt Creator 4.13.3 作为基础的原因：
- **经过生产验证** — Qt Creator 是全球数百万开发者使用的 IDE，其插件系统经过多年打磨，健壮可靠
- **架构清晰** — ExtensionSystem 库仅 27 个核心文件（约 5510 行），Aggregation 库 5 个文件（约 440 行），易于提取和理解
- **扩展性强** — Object Pool + Extension Point 机制提供了良好的插件间通信基础
- **Qt 原生** — 基于 Qt/C++，与目标平台（包括国产操作系统）高度兼容

## 二、项目五大目标

### 目标 1：插件化应用框架

**核心思路**：从 Qt Creator 提取插件系统骨架，构建独立的轻量级插件化应用框架。

**具体内容**：
- 提取 ExtensionSystem 库（插件管理器 PluginManager、插件基类 IPlugin、插件描述 PluginSpec、插件依赖解析等）
- 提取 Aggregation 库（对象聚合模式，支持跨组件 qobject_cast）
- 剥离 Qt Creator IDE 专有功能（编辑器、调试器、项目管理器等），只保留纯粹的插件加载/管理/生命周期机制
- 提供标准化的插件开发模板和接口定义
- 插件生命周期管理：loadLibrary → initialize → extensionsInitialized → delayedInitialize → aboutToShutdown

**交付标准**：
- 一个可独立编译运行的最小插件框架 App
- 能够动态扫描、加载、卸载插件（.dll/.so）
- 提供完整的插件开发 SDK 和示例插件

---

### 目标 2：可编辑可配置的页面 App

**核心思路**：页面以插件形式存在，应用界面完全由插件组合而成，支持运行时编辑和配置。

**具体内容**：
- 每个页面作为一个独立插件，遵循统一的页面插件接口
- 支持页面的动态添加、删除、排列、布局调整
- 页面内的控件/组件可通过配置文件（JSON/XML）定义，无需重新编译即可调整界面
- 提供页面设计器（可视化编辑模式），允许用户通过拖拽方式编排页面内容
- 支持页面间的导航和切换（Tab 页签、侧边栏导航、面包屑导航等）
- 支持主题和样式配置（基于 Qt Style Sheet 或自定义主题引擎）

**交付标准**：
- 页面插件可热加载（不重启应用即可切换/增加页面）
- 页面布局可通过配置文件描述
- 提供至少一种可视化的页面配置方式

---

### 目标 3：插件间数据交互

**核心思路**：插件之间有良好的数据通信机制，实现松耦合的模块间协作。

**具体内容**：
- **Object Pool 机制**（源自 Qt Creator）：插件通过 PluginManager::addObject() 注册服务对象，其他插件通过 PluginManager::getObject<T>() 获取，实现基于接口的服务发现
- **信号/槽通信**：利用 Qt 元对象系统，插件间通过信号槽进行异步事件通知
- **Extension Point（扩展点）模式**：定义扩展点接口，允许多个插件向同一扩展点注册实现（类似 Eclipse 的扩展点机制）
- **数据总线 / 事件总线**：提供全局的事件分发机制，支持广播式和定向式消息传递
- **共享数据模型**：支持插件间共享数据模型（如 QAbstractItemModel），一个插件提供数据，多个插件展示

**交付标准**：
- 插件 A 可以在不直接引用插件 B 的情况下，通过接口获取插件 B 的服务
- 提供完善的插件间通信 API 文档
- 包含通信机制的示例插件（发布者/订阅者模式演示）

---

### 目标 4：App 与外部数据交互

**核心思路**：应用具备与外部系统的数据对接能力，可作为数据展示/采集/控制终端。

**具体内容**：
- **网络通信**：支持 TCP/UDP Socket、HTTP/HTTPS 客户端请求（RESTful API 调用）
- **串口通信**：支持 RS-232/RS-485 串口数据收发（工业设备常用）
- **数据库对接**：支持通过 Qt SQL 模块连接 MySQL、PostgreSQL、SQLite 等数据库
- **文件数据交换**：支持 CSV、JSON、XML 等格式的数据导入/导出
- **消息队列**：预留与 MQTT、RabbitMQ 等消息中间件对接的能力
- **自定义协议**：提供协议抽象层，允许插件开发者实现自定义的通信协议
- **数据缓存与同步**：支持本地数据缓存，断网后自动重连和数据同步

**交付标准**：
- 至少支持 TCP/HTTP/串口 三种外部数据交互方式
- 外部数据接入以插件形式实现，可按需加载
- 提供数据交互接口的示例插件

---

### 目标 5：跨平台

**核心思路**：一套代码，多平台编译运行，覆盖 Windows 和主流国产 Linux 操作系统。

**目标平台矩阵**：

| 操作系统 | 版本 | 硬件架构 |
|----------|------|----------|
| Windows | Windows 7 / Windows 10 / Windows 11 | x86_64, x86_32 |
| 银河麒麟 | Kylin V10 | x86_64, Arm64 |
| 中标麒麟 | NeoKylin V7 | x86_64, Arm64 |
| Ubuntu | 18.04 / 20.04 / 22.04 | x86_64, x86_32, Arm64, Arm32 |

**具体内容**：
- 采用 qmake（.pro / .pri）构建系统，已移除 CMake 和 QBS 构建描述，确保构建系统统一
- 所有平台特定代码通过 Qt 跨平台 API 封装（QDir、QProcess、QSettings 等）
- 保留 Windows 特定条件编译块（win32:、msvc:）和 Linux 特定条件编译块（linux-*）
- 插件以动态库形式加载（Windows: .dll, Linux: .so），路径配置通过环境变量或配置文件管理
- UI 适配不同分辨率和 DPI（High-DPI 支持）
- 针对国产操作系统进行兼容性测试和必要的适配

**交付标准**：
- 同一套源码可在所有目标平台编译通过
- 插件在不同平台上行为一致
- 提供各平台的编译部署脚本和文档

## 四、技术架构概览

### 4.1 插件系统核心架构

```
┌─────────────────────────────────────────────────┐
│                   应用入口 (main.cpp)              │
│  创建 QApplication → 实例化 PluginManager          │
│  → 扫描插件目录 → 加载插件 → 启动事件循环            │
└────────────────────┬────────────────────────────┘
                     │
        ┌────────────▼────────────┐
        │    PluginManager        │
        │  (ExtensionSystem 库)    │
        │                         │
        │  ● 插件扫描与发现        │
        │  ● 依赖关系解析          │
        │  ● 生命周期管理          │
        │  ● Object Pool 管理     │
        │  ● Extension Point 注册 │
        └────────────┬────────────┘
                     │
    ┌────────────────┼────────────────┐
    │                │                │
┌───▼───┐      ┌────▼────┐      ┌───▼───┐
│Plugin A│      │Plugin B │      │Plugin C│
│(.dll/  │◄────►│(.dll/   │◄────►│(.dll/  │
│ .so)   │      │ .so)    │      │ .so)   │
│        │      │         │      │        │
│ IPlugin│      │ IPlugin │      │ IPlugin│
└───┬───┘      └────┬────┘      └───┬───┘
    │                │                │
    └────────────────┼────────────────┘
                     │
        ┌────────────▼────────────┐
        │    Aggregation 库       │
        │  对象聚合 + 跨组件查询    │
        └─────────────────────────┘
```

### 4.2 插件生命周期

```
应用启动
  │
  ├─ 1. loadLibrary()          ← 加载动态库，读取 JSON 元数据
  │
  ├─ 2. initialize()           ← 插件初始化（注册服务、创建核心对象）
  │     (IPlugin::initialize)
  │
  ├─ 3. extensionsInitialized() ← 扩展初始化完成（可安全访问其他插件服务）
  │     (逆序调用)
  │
  ├─ 4. delayedInitialize()    ← 延迟初始化（非关键的后台任务，20ms 定时器）
  │
  ├─ ... 应用正常运行 ...
  │
  └─ 5. aboutToShutdown()      ← 应用关闭前通知（释放资源、保存状态）
```

### 4.3 插件间通信机制

| 机制 | 说明 | 适用场景 |
|------|------|----------|
| Object Pool | addObject/getObject 服务注册与发现 | 插件提供服务给其他插件使用 |
| Extension Point | 多个插件向同一扩展点注册实现 | 菜单项、工具栏按钮、页面贡献 |
| 信号/槽 | Qt 原生异步事件通知 | 状态变化、用户操作响应 |
| Aggregation | 对象聚合，跨组件 qobject_cast | 复合对象的接口查询 |

## 八、关键术语

| 术语 | 说明 |
|------|------|
| **HldPlugin** | 本项目名称，源自 Qt Creator 重命名后的标识 |
| **ExtensionSystem** | Qt Creator 的插件管理框架库，是本项目的核心基础 |
| **Aggregation** | Qt Creator 的对象聚合库，支持将多个 QObject 组合为逻辑实体 |
| **PluginManager** | 插件管理器，负责插件的扫描、加载、生命周期管理 |
| **IPlugin** | 插件基类，所有插件必须继承此类 |
| **PluginSpec** | 插件描述/元数据，从插件 JSON 文件中读取 |
| **Object Pool** | 对象池，PluginManager 管理的全局对象注册表 |
| **Extension Point** | 扩展点，定义插件可贡献功能的接口 |
| **CorePlugin** | 核心插件，应用启动的必要插件，提供基础 UI 框架 |
| **Plugin IID** | 插件接口标识符（org.qt-project.Qt.QtHldpluginPlugin） |
